## 재현 → 진단 (추측 금지, 직접 원시출력 측정)
1. **qwen3:14b 직접 호출** (동일 시스템 프롬프트 + temp 0.1): 두 프롬프트 모두 **완벽한 JSON** 반환, `parsed_ok: true`. → "qwen3:14b가 멍청하다" 가설 **반증**.
2. 파이프라인은 `settings.llm_main_model`이 아니라 `runtime.get_chat_model()`을 사용 (`intent_parser.py:_ollama_model_for`). 실제 파일 `backend/data/runtime_settings.json` 확인:
   ```json
   { "chat_model": "huihui_ai/qwen3-vl-abliterated:8b" }
   ```
3. **VL 8B 모델 직접 호출**: 반환값이 **빈 문자열 `''`** (깨진 JSON이 아니라 출력 자체가 없음) → `_extract_json('')` = None → 폴백.

## 원인
- runtime chat_model이 의도해석에 **부적합한 8B VL(비전) abliterated 모델**로 설정돼 있었음.
- VL 모델은 텍스트 전용 intent JSON 프롬프트에 빈 출력을 냄.
- 폴백 모델(`huihui_ai/qwen3-abliterated:14b`, EXPLICIT용)이 **동작은 성공시켜서** 증상이 오직 "느림"으로만 표출됨 (매 요청 LLM 2회 호출).
- 경고 로그가 **실패한 모델명을 안 남겨** 오진을 유발함.

## 수정
1. `backend/data/runtime_settings.json` → `chat_model: "qwen3:14b"` 복원 (실질 수정).
2. `backend/app/api/routes_models.py` — `is_chat_compatible()` 추가: `embed`/`vl` 포함 모델을 채팅 후보에서 제외. GET 목록 필터 + PUT `/models/chat` 검증 양쪽에 적용 (API 직접 호출로도 부적합 모델 설정 차단).
3. `backend/app/pipeline/intent_parser.py` — 폴백 경고에 **실패 모델명 + 출력 길이** 기록.
4. `backend/cli/trace.py` — `_make_orchestrator` 스텁(NotImplementedError)을 실제 `deps` 연결로 교체. 이 dry-run 도구가 위 진단을 가능케 함.
