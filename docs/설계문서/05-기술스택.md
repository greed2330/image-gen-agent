# 05. 기술 스택

> 최종 수정: 2026-06-10
> ⚠️ 언어/프레임워크는 **사용자 최종 확정 필요**. 아래는 설계 기반 권장안.

## 권장 스택 (Primary 추천)

| 레이어 | 선택 | 이유 |
|--------|------|------|
| **백엔드 / 오케스트레이터** | **Python + FastAPI** | ComfyUI·ollama·mem0 모두 Python 생태계 1급 시민. 이미지/텐서 처리, ML 글루 코드에 자연스러움. mem0 Python SDK가 가장 성숙. |
| **프론트엔드 (채팅 UI)** | **Next.js (React) + TypeScript** | Gemini/Claude/GPT급 채팅 UX 구현에 생태계 최강. 스트리밍·이미지 첨부·갤러리 컴포넌트 풍부. |
| **실시간 통신** | WebSocket (생성 진행률 스트리밍) + REST | ComfyUI도 WebSocket으로 진행률 push |
| **상태/DB** | SQLite (채팅방·메시지 메타) + Qdrant (mem0 벡터) | 단일 사용자라 SQLite로 충분. 무겁게 안 감 |
| **이미지 엔진** | ComfyUI (별도 프로세스, HTTP/WS API) | 동적 워크플로우 조립 |
| **LLM 런타임** | ollama (로컬) + 클라우드 API SDK | 하이브리드 |

### 왜 Python 백엔드 + JS 프론트 (풀스택 단일언어 대신)

- 이미지 생성 글루(ComfyUI 워크플로우 조립, 이미지 후처리, mem0)는 Python이 압도적으로 편함.
- 채팅 UI 완성도는 React 생태계가 압도적.
- 둘을 HTTP/WS로 분리하면 각자 강점 영역 사용 + 책임 분리 명확.
- 단점: 두 언어 관리. 단일 사용자 로컬 프로젝트라 운영 부담은 작음.

## 대안 스택 (참고)

| 대안 | 장점 | 단점 |
|------|------|------|
| **풀 TypeScript** (Next.js + Node 백엔드, ComfyUI/ollama는 HTTP) | 단일 언어 | mem0/이미지 글루가 Python보다 불편. ComfyUI 커스텀 연동 시 레퍼런스 적음 |
| **풀 Python** (백엔드 + Gradio/Streamlit UI) | 단일 언어, 빠른 프로토타입 | 채팅 UX 고도화(업계급)에 한계. 커스텀 UI 어려움 |
| 기존 오픈소스 채팅 UI(OpenWebUI 등) 포크 | 빠른 시작 | 우리 오케스트레이션 로직 끼워넣기 복잡, 학습 비용 |

## 권장 디렉토리 구조 (구현 시작 시)

```
image-gen-agent/
├─ backend/                 # Python FastAPI
│  ├─ orchestrator/         # 두뇌: intent/router/compiler/resolver/critic
│  ├─ vram/                 # VRAM 매니저 (ollama/comfyui 로드·언로드)
│  ├─ comfy/                # ComfyUI 워크플로우 빌더 + 클라이언트
│  ├─ memory/               # mem0 래퍼
│  ├─ llm/                  # ollama + cloud API 라우팅
│  └─ api/                  # REST/WS 엔드포인트
├─ frontend/                # Next.js
│  ├─ app/                  # 채팅 화면
│  └─ components/           # 채팅방·메시지·갤러리
├─ docs/                    # 설계/작업기록/기타 (현재 폴더)
├─ workflows/               # ComfyUI 템플릿 그래프(JSON)
└─ config/                  # 모델 경로·프리셋·라우팅 규칙
```
