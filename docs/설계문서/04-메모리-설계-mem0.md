# 04. 메모리 설계 (mem0)

> 최종 수정: 2026-06-10

## 왜 mem0인가 (vs 단순 RAG)

- mem0는 애플리케이션과 LLM 사이에 앉는 **지능형 메모리 레이어**. add/search/update를 단순 API로 추상화.
- 2026 SOTA는 **하이브리드 메모리**: 벡터 검색(비정형) + 그래프(정형 관계) + 에피소드(시간 순서). mem0가 이 방향.
- 정확도: 그래프 강화 메모리 ~90% vs 단순 RAG ~60% (Cognee 벤치 인용).
- 2026-04 mem0는 토큰 효율적 메모리 알고리즘(단일 패스 계층 추출 + 멀티시그널 검색) 릴리스.
- 단점: 내부 저장/검색 가시성이 낮음 → 디버깅 시 답답할 수 있음. 그래서 **로그/덤프 가능하게 래핑**할 것.

## 이 프로젝트에서 메모리가 담는 것

| 종류 | 예시 | 스코프 |
|------|------|--------|
| **사용자 취향(전역)** | "이 사람은 항상 1.5:1 세로 구도 선호", "그림체는 부드러운 셀화 선호", "특정 LoRA 자주 씀" | user 전역 |
| **채팅방 컨텍스트** | 그 방에서 작업 중인 캐릭터 설정·세계관·이전 생성 의도 | session(채팅방)별 |
| **성공 레시피** | "이 프롬프트+파라미터 조합이 좋았다" → 재사용 | user 전역 + 태그 |
| **레퍼런스 자산** | 자주 쓰는 레퍼런스 이미지/캐릭터 시트 | user/session |

## 채팅방별 기억 분리

- 업계 채팅 UX처럼 **채팅방 생성/삭제/전환** 지원.
- mem0의 스코프 키 활용: `user_id`(전역 취향) + `run_id`/`session_id`(채팅방별).
- **2계층 검색**: 의도 해석 시 (1) 현재 채팅방 메모리 + (2) 전역 사용자 취향을 함께 검색해 컨텍스트 구성.
- 채팅방 삭제 시 해당 `session_id` 메모리만 삭제, 전역 취향은 보존.

## 자가 호스팅 구성 (전부 로컬)

mem0는 OSS로 자가 호스팅 가능. 백엔드를 전부 로컬로:

```
mem0 (OSS)
 ├─ 벡터 스토어: Qdrant (로컬) 또는 Chroma
 ├─ 그래프 스토어: (선택) Neo4j 로컬 — 관계형 정확도 필요 시
 ├─ 임베딩: ollama 로컬 임베딩 모델 (예: nomic-embed-text, bge-m3)
 └─ LLM(메모리 추출용): 로컬 ollama 모델 재사용
```

> 임베딩까지 로컬로 두면 메모리 파이프라인 전체가 외부 유출 0. NSFW 컨텍스트도 안전.

## 메모리 쓰기/읽기 흐름

```
[대화/생성 종료]
   └─> mem0.add(메시지+결과+피드백, user_id, session_id)
        (mem0가 자동으로 "기억할 만한 사실" 추출·저장)

[새 의도 해석 시작]
   └─> mem0.search(현재 발화, user_id, session_id, limit=N)
        └─> 검색된 취향/컨텍스트를 Intent Parser 프롬프트에 주입
```
