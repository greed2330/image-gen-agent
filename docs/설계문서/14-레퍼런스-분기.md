# 14. 레퍼런스 분기 (IPAdapter / ControlNet / img2img)

> 상태: 설계 (2026-06-24). 구현 전 오너 확인 필요 항목은 §7 참조.
> 관련: Doc 11(img2img), 체크리스트 「Phase 3」·「알려진 한계」.

---

## 1. 목표

채팅 + 레퍼런스 이미지로 **"이 캐릭터로 / 이 포즈로 / 이런 느낌으로"** 를 자동 분기해 처리한다.

- **현재 한계:** 레퍼런스가 있으면 **무조건 img2img(통째 변형)**. → 캐릭터만 옮기거나 포즈만 옮기는 걸 못 함.
- **이번 작업:** 레퍼런스 활용 의도를 3가지로 갈라 각각 다른 ComfyUI 워크플로로 보낸다.

---

## 2. 핵심 개념 — "레퍼런스에서 **무엇을 KEEP** 하나"

레퍼런스 1장에서 가져올 수 있는 건 ① 캐릭터 외형, ② 포즈/구도, ③ 그림체. 분기는 무엇을 유지하느냐로 갈린다.

| reference_mode | 레퍼런스에서 KEEP | 텍스트로 CHANGE | 사용자 표현 예 |
|---|---|---|---|
| **character** (IPAdapter) | 캐릭터 외형·정체성·분위기 | 포즈·배경·행동 | "이 **캐릭터**를 ~하게" |
| **pose** (ControlNet) | 포즈·골격·구도 | 캐릭터 외형·색 | "이 **포즈/자세**로 ~한 캐릭터를" |
| **vary** (img2img) | 전체 통째(구도+색+외형) | 약한 변형 | "이 이미지를 ~느낌으로 살짝" |

> character와 pose는 **거울쌍**이다: 무엇을 유지하고 무엇을 바꾸는지가 정반대.
> "그림체 그대로"는 대개 **동일 체크포인트**가 처리하므로 별도 스타일 이식이 꼭 필요하진 않다.

---

## 3. 분기 결정 — 방식 A (텍스트 의도)

**분기는 사용자 말투(텍스트)로 결정한다.** `reference_mode`(character/pose/vary)를 기존 텍스트 LLM(qwen3)이 intent 파싱 시 함께 분류.

### 왜 텍스트인가 (방식 A 채택, 방식 B[qwen3-vl] 기각)
- "무엇을 KEEP할지"는 **이미지가 아니라 사용자 의도**에 있다. 같은 이미지라도 "이 캐릭터를~"이면 character, "이 포즈로~"면 pose.
- IPAdapter는 이미지를 CLIP 임베딩으로, ControlNet은 전처리맵으로 **직접 소비** — 이미지 *설명*이 워크플로에 쓰이지 않음. 즉 qwen3-vl 출력이 소비되지 않는다.
- VRAM: qwen3-vl(8B ~6GB) 분기마다 로드/언로드는 단일점유 규칙상 스왑 오버헤드.
- **버린 대안 B:** qwen3-vl 비전 분기 — 위 이유로 분기엔 불필요. qwen3-vl은 추후 "레퍼런스→캐릭터 태그 자동 추출"이라는 **다른 역할**로만 등판 가능(분기자가 아님).

### 분류 규칙 (한국어 표현 → mode)
- **character**: "이 캐릭터/이 사람/얘를 ~", 외형 유지 + 새 행동/포즈/배경 지시
- **pose**: "이 포즈/자세/구도로 ~", "이 자세 그대로 ~한 캐릭터"
- **vary**: "이 이미지를 ~느낌으로", "살짝 바꿔/변형", 또는 **모호**할 때 기본값

### 모호 시 기본값 = vary (img2img)
- 이유: img2img는 원본을 가장 보존(denoise 0.55)하므로 **오분류 시 피해가 가장 작다**. character/pose로 잘못 가면 원본과 동떨어진 결과가 나옴.
- 대안(되물어보기)은 채팅 흐름을 끊으므로 1차 범위 밖. 기본값으로 처리.

## 4. 기능 어프로치 (파이프라인 변경)

```
① 사용자: 이미지 업로드 + 메시지
② IntentParser: 기존 태그 + reference_mode 분류 (system prompt 규칙+예시)
③ WorkflowRouter: reference 있음 + reference_mode →
     character → WorkflowType.IPADAPTER
     pose      → WorkflowType.CONTROLNET
     vary      → WorkflowType.IMG2IMG (기존)
④ orchestrator._build_workflow: 분기별 템플릿 채우기
     IPAdapter   → ipadapter.json   (레퍼런스 → IPAdapterModelLoader/Apply)
     ControlNet  → controlnet.json  (레퍼런스 → 전처리 → ControlNet Apply)
     img2img     → img2img.json     (기존)
⑤ 생성 (기존 generate + 진행률 스트리밍 그대로)
```

### 코드 변경 지점
- `schemas.py`: `Intent.reference_mode: ReferenceMode`(enum, 기본 vary). `WorkflowType`에 이미 IPADAPTER/CONTROLNET enum 존재.
- `intent_parser.py`: reference_mode 추출(레퍼런스 있을 때만 의미), system prompt 규칙/예시.
- `workflow_router.py`: `intent.reference` 있으면 reference_mode로 3분기(현재는 무조건 IMG2IMG).
- `workflows/ipadapter.json`, `workflows/controlnet.json` 신규.
- `orchestrator._build_workflow`: IPADAPTER/CONTROLNET 템플릿 분기 + 레퍼런스 슬롯 채우기.
- 업로드는 기존 `comfyui.upload_image` 재사용(이미 input dir에 write).

## 5. ComfyUI 노드 현실 (2026-06-24 실측) — 설계 제약

| 자산 | 상태 |
|---|---|
| IPAdapter_plus 노드 (IPAdapter/Advanced/ModelLoader 등) | ✅ 설치됨 |
| ip-adapter-plus_sdxl_vit-h / plus-face | ✅ 설치됨 |
| CLIP-ViT-H | ✅ 설치됨 |
| ControlNet union promax 모델 | ✅ 설치됨 |
| **포즈 전처리기 (comfyui_controlnet_aux / DWPose / OpenPose)** | ❌ **미설치** |
| Canny (코어 노드) | ✅ 사용 가능 |

**핵심 제약:** ControlNet은 입력 이미지를 **제어맵(control map)으로 전처리**해야 한다.
- **canny(윤곽)**: 코어 `Canny` 노드로 전처리기 설치 없이 가능 → "윤곽/구도 유지" 수준의 pose 분기 가능.
- **openpose/DWPose(골격 포즈)**: 전용 전처리기(comfyui_controlnet_aux) **설치 선행 필요** → 의존성 추가 결정 사항.

→ **character(IPAdapter)와 vary(img2img)는 즉시 구현 가능.** pose는 전처리기 수준에 따라 갈림(§6).

---

## 6. 단계 제안

- **3a (신규 설치 없이):** character(IPAdapter) + vary(img2img) + pose=**canny**(윤곽/구도 유지).
  - 즉시 빌드 가능. "이 캐릭터로"가 핵심 가치라 우선순위 높음.
- **3b (comfyui_controlnet_aux 설치 후):** pose를 **DWPose/OpenPose**(정밀 골격)로 승격.
  - 의존성 추가 → 이유·대체불가성 기록 후 진행(오너 확인).
