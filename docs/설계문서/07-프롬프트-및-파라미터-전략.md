# 07. 프롬프트 & 파라미터 전략 (핵심)

> 최종 수정: 2026-06-10
> 이 프로젝트의 성패를 가르는 부분. ②번 "모델 포맷에 맞는 프롬프트 합성"과 ③번 "능동적 파라미터/스타일 조정"의 실제 설계.

## 0. 전제: 모든 것은 "어떤 체크포인트냐"에 종속

| 층 | 선택 | 포맷 |
|----|------|------|
| 실행 엔진 | ComfyUI | (AI 아님, 런타임) |
| 이미지 생성 AI = 체크포인트 | **Illustrious/NoobAI (SDXL)** 메인 | **danbooru 태그** |
| 〃 | Chroma/Flux 보조 | **자연어 문장** |

→ Prompt Compiler / Param Resolver는 **체크포인트별로 분기**한다. 아래는 Illustrious(태그형) 중심, Flux(자연어형)는 별도 프로파일.

---

## 1. 핵심 설계 원칙 (저점 보장 + 고점)

1. **LLM은 raw 수치를 자유 생성하지 않는다.** 검증된 프리셋을 *선택*하고, 허용 범위 안에서만 *미세 조정(nudge)*. 하드 클램프로 쓰레기값 차단.
2. **저점 = 프리셋.** LLM이 아무 판단을 못 해도 모델/스타일 프리셋만으로 괜찮은 결과.
3. **고점 = LLM의 정보 기반 조정.** 그림체에 맞는 LoRA·샘플러·태그 가중치 선택 + Critic 루프.
4. **태그는 검증한다.** LLM이 지어낸 비존재 태그는 사전(allowlist)으로 걸러내거나 보정.
5. **결정은 구조화 출력(JSON)으로.** LLM은 자연어가 아니라 기계가 파싱할 JSON을 뱉는다.
6. **이해와 합성을 분리한다.** "한국어 의도 파악"(범용 LLM)과 "danbooru 프롬프트 합성"(전용 모델 TIPO)은 별개 작업. 범용 LLM은 거친 씨앗 태그만 뽑고, 정밀 합성·조합·정렬은 TIPO에 위임. → LLM의 어휘 부담을 덜어 hallucination을 구조적으로 낮춤.

## 1-A. 합성 파이프라인 (레이어 분리 + TIPO)

> 이 프로젝트 hallucination 대책의 핵심. 2026-06-10 결정(오너 제안 → TIPO 도입).

```
[A. 이해]  한국어 발화 → 거친 의도 + 씨앗 태그
            담당: qwen3:14b / 클라우드 (SFW) · abliterated (NSFW)
            예: "분홍 머리 트윈테일 소녀가 웃으며" → ["1girl", "pink hair", "twintails", "smile"]
        │
        ▼
[B. 합성]  씨앗 태그 → 완성된 danbooru 프롬프트 (확장·조합·정렬)
            담당: TIPO (KGen, danbooru 전용 학습, 200M/500M)
            - 연관 태그 보강, 태그 co-occurrence·순서를 모델이 앎
            - SDXL(태그) = TIPO 태그 모드 / Chroma(자연어) = TIPO NL 모드
        │
        ▼
[C. 검증]  allowlist 대조 → 비존재 태그 제거/치환 (결정론적 안전망)
            담당: 코드 (danbooru 태그 CSV)
            - TIPO는 "소프트" 감소, 이 단계가 "하드" 0% 보장
        │
        ▼
[이미지 모델]  Illustrious/NoobAI 등
```

**왜 이 구조인가 (검증/추론 구분):**
- ✅ **검증된 사실**: TIPO/DanTagGen은 danbooru 태그로 학습된 전용 프롬프트 생성 모델(ICLR 2026). Illustrious/NoobAI도 danbooru 태그 학습 → **둘이 같은 어휘(공용어)를 씀.** 별도 포맷 정의 불필요. ComfyUI 노드 존재.
- ✅ **구조적 기대**: TIPO가 붙이는 태그는 danbooru 분포에서 나오므로 비존재 태그 확률이 범용 LLM보다 낮음. + allowlist로 최종 0% 보장.
- ⚠️ **미검증(과신 금지)**: TIPO의 **NSFW 태그 처리 품질**은 문서로 확인 안 됨. danbooru 학습이라 explicit 태그를 "알 것"이라는 합리적 추론일 뿐. **Phase 1/2 실측 전까지 확정 아님.**
- ⚠️ **미검증**: 언더스코어/공백·괄호 이스케이프 등 표기 관례가 TIPO ↔ 체크포인트 간 100% 일치하는지. ComfyUI 노드가 흡수할 가능성 높으나 Phase 1에서 실측.
- ⚠️ **한계**: TIPO는 "이해"는 못 함(통계적 확장기). 앞단 A 레이어(범용 LLM)는 여전히 필수. Chroma는 자연어라 SDXL과 경로 분리.

**A 레이어 내부의 SFW/NSFW 분리 (설계 구멍 보완):**
> session_002에서 발견. A 레이어의 기본 담당 `qwen3:14b`는 **검열 모델**이라 NSFW 발화를 파싱하다 거부할 수 있다. 분리 원칙이 없으면 무검열 요청이 앞단에서 막힌다.

- **원칙 1 — 임상적 핸드오프(clinical handoff)**: `qwen3:14b`에게 "흥분시키는 묘사 생성"이 아니라 "발화를 구조화·분류"하는 역할로 프레이밍한다. 분류/임상 톤은 검열 모델의 거부율을 크게 낮춘다(검열 모델도 "생성"은 거부해도 "분류"는 대체로 응함).
- **원칙 2 — 에스컬레이션(폴백)**: `qwen3:14b`가 임상 분류조차 거부하는 등급만 `abliterated:14b`가 떠안는다. 즉 abliterated는 "기본"이 아니라 **거부 시 폴백**. (저능 트레이드오프 노출 최소화)
- **원칙 3 — 동일 출력 계약**: 어느 모델이 담당하든 출력은 같은 씨앗 태그 JSON. 다운스트림(B 합성·C 검증)은 누가 파싱했는지 모른다 → 분리는 A 레이어 내부에 캡슐화.
- ⚠️ **미확정(경계선)**: `qwen3:14b`가 어느 explicit 등급에서 임상 분류조차 거부하는지는 문서로 알 수 없음. **Phase 2 실측**으로 확정. 그 경계가 곧 두 모델의 분담선.

## 2. ② 프롬프트 합성 — 시스템프롬프트 골격 (Illustrious/태그형)

> ⚠️ TIPO 도입(§1-A)으로 **LLM의 부담이 줄었다.** 아래 스키마의 `positive_tags`는 이제 "완성본"이 아니라 **TIPO에 넘길 씨앗(seed)**. 태그 순서·연관 보강·co-occurrence는 TIPO가 처리하므로, 시스템프롬프트의 "태그 순서 규약/대량 사전 주입" 비중은 낮아진다(없애지는 않음 — 씨앗 품질이 합성 품질을 좌우).
> **시스템 프롬프트는 더 이상 "정수"가 아니라 부품.** 정수는 "레이어 분리 + TIPO + allowlist" 아키텍처.

### LLM 출력 스키마 (JSON 강제)
```json
{
  "model_profile": "illustrious",
  "positive_tags": ["1girl", "solo", "..."],
  "negative_tags": ["..."],
  "tag_weights": {"specific_tag": 1.2},
  "aspect": "portrait",
  "style_preset": "soft_cel",
  "loras": [{"name": "xxx", "weight": 0.7}],
  "workflow": "txt2img",
  "reference_use": null
}
```

### 시스템프롬프트에 반드시 박을 것
- **출력 스키마**: 위 JSON만 출력. 설명 금지.
- **태그 순서 규약**: `품질토큰 → 인원수(1girl/1boy/solo) → 캐릭터/고유명 → 외형(머리·눈·체형) → 의상 → 동작/포즈/표정 → 배경 → 구도/연출 → 스타일/품질 보정`.
- **품질토큰 프리셋**(모델별로 다름! NoobAI ≠ Pony): positive 기본 + negative 기본을 주입.
- **가중치 문법**: `(tag:1.2)` 형태, 0.6~1.4 권장 범위, 남발 금지.
- **태그 사전 참조 규칙**: "아래 제공된 태그/캐릭터 목록에 있는 것만 사용. 없으면 가장 가까운 일반 태그로." (RAG/주입)
- **한국어→태그 매핑 예시 몇 개**(few-shot): "분홍 머리 트윈테일 소녀가 웃으며" → `1girl, solo, pink hair, twintails, smile`.

### 태그 정확도 보정 (hallucination 방지)
- danbooru 태그 CSV(autocomplete용으로 흔히 쓰이는 것)를 **allowlist**로 보유.
- LLM 출력 태그를 사전과 대조 → 없는 태그는 (a) 임베딩 최근접 태그로 치환 또는 (b) 드롭 + 경고 로그.
- 캐릭터는 `캐릭터명 → 정식 태그 + 부속 외형 태그` 매핑 테이블 별도 유지.
- 레퍼런스 이미지가 있으면 qwen3-vl-abliterated:8b로 태그 추출 → positive_tags 시드로.

> **⚠️ 실측 발견 (2026-06-18, Phase 1) — TIPO 검열 태그 자동 주입.** TIPO-500M-ft에 NSFW 씨앗(예: `nude, lying on bed`)을 주면 danbooru 분포 특성상 `censored`, `mosaic censoring`, `bar censor` 류 태그를 통계적으로 끼워넣음. 그대로 positive로 보내면 **무검열 프로젝트인데 모자이크가 생성됨.** → prompt_compiler ③ allowlist 단계에서 **검열 태그 차단리스트(`censored`/`mosaic censoring`/`bar censor`/`convenient censoring` 등)를 positive에서 제거하고 negative로 강제 이동**해야 함. 이 처리는 무검열 출력의 필수 안전망이며, allowlist 단계가 hallucination 제거뿐 아니라 검열태그 제거 역할도 겸함을 실증.

### Flux/자연어형 프로파일 (분기)
- 출력은 태그가 아니라 **서술형 문장**(장면·조명·카메라·질감 묘사).
- 품질토큰/가중치 문법 거의 불필요. 네거티브 개념도 약함.
- 같은 intent라도 `model_profile`에 따라 Compiler가 다른 시스템프롬프트를 사용.

## 3. ③ 파라미터 & 스타일 — 능동 조정 메커니즘

### 3-1. 모델 프리셋 (저점의 근거)
체크포인트마다 모델카드 권장값을 config로 보유:
```yaml
illustrious_base:
  steps: 28
  cfg: 5.0
  sampler: "Euler a"        # 또는 DPM++ 2M
  scheduler: "normal"        # 또는 karras
  clip_skip: 2
  resolutions:
    portrait: [832, 1216]
    landscape: [1216, 832]
    square: [1024, 1024]
  quality_pos: "masterpiece, best quality, amazing quality"
  quality_neg: "worst quality, low quality, jpeg artifacts, ..."
  allowed_nudge: { cfg: [-1.0, 1.0], steps: [-8, 12] }   # LLM 조정 허용폭
  hard_clamp:   { cfg: [3.0, 9.0], steps: [12, 50] }     # 절대 한계
```

### 3-2. 스타일(그림체) 프리셋 — 모델 위에 레이어
```yaml
soft_cel:                    # 부드러운 셀화
  extra_pos: "soft lighting, pastel colors, smooth shading"
  loras: [{name: "soft_anime_style", weight: 0.6}]
  cfg_adjust: -0.5
sharp_detailed:             # 쨍한 디테일
  extra_pos: "highly detailed, sharp focus, intricate details"
  cfg_adjust: +0.5
  steps_adjust: +6
watercolor:
  extra_pos: "watercolor (medium), traditional media"
  loras: [{name: "watercolor", weight: 0.7}]
```
→ 그림체별로 "어떤 옵션을 어느 정도로"가 여기 박힘. LLM은 `style_preset` 이름만 고르면 됨.

### 3-3. 결정 주체 분담 (누가 무엇을 정하나)
| 항목 | 결정 주체 | 방식 |
|------|----------|------|
| 모델/체크포인트 선택 | LLM (Router) | intent의 도메인(애니/실사)으로 |
| 워크플로우(txt2img/i2i/controlnet/ipadapter) | LLM (Router) | 레퍼런스 유무·의도로 |
| 태그/프롬프트 내용 | LLM (Compiler) | §2 스키마 + 사전 검증 |
| 스타일 프리셋·LoRA 선택 | LLM (Compiler) | 프리셋 목록 중 선택 |
| 기본 파라미터 | **프리셋(결정론적)** | 모델 프리셋이 기본값 |
| 파라미터 미세조정 | LLM (Resolver) | `allowed_nudge` 범위 내에서만 |
| 최종 안전 클램프 | **코드(결정론적)** | `hard_clamp`로 강제 |
| 재시도 판단 | LLM (Critic) | qwen3-vl-abliterated:8b 검사 후 |

### 3-4. 조정 흐름 (예)
```
intent: "몽환적인 분위기로 부드럽게"
 → Router: 애니 → illustrious_base
 → Compiler: style_preset=soft_cel, dreamy 관련 태그 추가, lora 선택
 → Resolver: illustrious_base 프리셋 로드
            + soft_cel의 cfg_adjust(-0.5) 적용 → cfg 4.5
            + LLM nudge: "몽환 → steps 약간↑" steps +6 (allowed_nudge 내)
            + hard_clamp 검사 통과
 → ComfyUI 그래프 조립 → 생성
```
