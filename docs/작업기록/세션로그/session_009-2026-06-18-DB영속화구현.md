## 주요 결정 (무엇을·왜·버린 대안)

### 09-1단계: PromptCompiler 보호 메커니즘
- **identity_tags에 (tag:1.3) 가중치 부여** — Danbooru 가중치 문법. TIPO 확장 태그(~23개)에 수적으로 밀리는 문제를 emphasis로 보완
- **속성군 충돌 → negative 강제 이동** — `tag_attribute_groups.yaml` 정의: hair_color/eye_color/fur_color. identity가 한 슬롯을 선점하면 TIPO가 추가한 같은-군 태그는 negative로
- **버린 대안**: negative 전체 목록 하드코딩 → 유지보수 불가. YAML 속성군 정의로 선택

### 09-2단계: Intent 슬롯 분리
- `identity_tags` (WHO, 영속) / `scene_tags` (WHAT, 이번 요청만)로 분리
- **TIPO 입력**: `subjects + scene_tags`만 — identity_tags는 TIPO에 넘기지 않음. TIPO가 identity를 확장하면 오히려 모순 태그 추가 위험
- **`seed_tags` property**: 하위 호환용. `identity_tags + scene_tags` 반환
- **tag_groups.py 모듈 추출**: `prompt_compiler`와 `intent_parser` 둘 다 normalize/group 로직 필요 → CLAUDE.md "No duplicate logic" 원칙으로 서비스 레이어 추출

### 10단계: SQLite 영속화
- **SQLModel 선택** — Pydantic 네이티브 통합 + 추후 PostgreSQL 마이그레이션은 연결 문자열만 교체
- **FK 선언 없이 수동 cascade** — SQLite는 FK 강제 기본 OFF; PRAGMA로 켜는 대신 `delete_room()`에서 수동으로 Message/Generation 먼저 삭제. 이유: PRAGMA 설정 누락 시 데이터 고아 발생 위험이 명시적 코드보다 큼
- **Room.identity_tags: JSON 컬럼** — 캐릭터 카드 누적. `merge_identity()`로 같은 속성군 슬롯은 덮어씀(예: white hair → red hair), 무관한 태그는 append
- **seed를 `_build_workflow` 밖으로 이동** — Generation 레코드에 seed 저장을 위해 orchestrator 레벨에서 생성 후 주입

### Doc 13: 다운로드 버튼 (소규모)

- 프론트 라이트박스에 ⬇ 다운로드 버튼 추가
