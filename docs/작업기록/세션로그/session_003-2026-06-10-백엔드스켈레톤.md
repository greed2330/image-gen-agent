## 주요 결정 (무엇을·왜·버린 대안)

### 1. TIPO/DanTagGen 도입 확정 (Opus)

**오너 질문**: "이미지 생성 프롬프트 합성 전용 스페셜리스트 모델이 있나?"

→ **있음. TIPO (KGen, ICLR 2026) / DanTagGen.**

**검증된 사실 (GitHub 원문 확인)**:
- TIPO 200M/500M, danbooru + 자연어 학습, ICLR 2026 발표 확인
- Illustrious/NoobAI도 danbooru 태그 학습 → **같은 공용어. 포맷 정의 불필요.**
- ComfyUI 노드 실존

**⚠️ 이 세션 교훈 — 과신 경계**: Opus가 처음에 "TIPO가 NSFW native 처리, hallucination 0"이라 단정 → 오너 "팩트야?" 지적 → 재확인. NSFW 품질·표기관례 일치는 **문서 미확인 = 미검증**. Phase 1/2 실측 전 확정 금지.

**왜 도입**: 범용 LLM의 어휘 부담↓ → hallucination 구조적 감소. allowlist 검증과 합쳐 사실상 해결. 시스템 프롬프트는 "정수"에서 "부품"으로 격하.

**버린 대안**: 수동 NSFW 태그 사전(거대 시스템 프롬프트 주입) → TIPO가 학습된 사전 그 자체이므로 불필요. Simplicity First.

### 2. 합성 파이프라인 3단 구조 확정 (§1-A)

```
[A 이해] 한국어 → 씨앗 태그 (qwen3:14b / abliterated)
    ↓
[B 합성] TIPO: 씨앗 → 완성 danbooru 프롬프트 (확장·조합·정렬)
    ↓
[C 검증] allowlist: 비존재 태그 제거 (결정론적 안전망)
```

**NSFW 레이어 분리**: qwen3:14b는 "분류·구조화"만(거부율↓), 막히는 등급만 abliterated가 이해까지 떠안음. 경계선은 Phase 2 실측.

### 3. CLI 디버그 하버스 추가 (오너 요청)

**오너 요청**: "에이전트가 직접 터미널로 테스트해볼 수 있어야 어느 단계에서 문제가 생겼는지 알 수 있다"

→ `cli/trace.py` (`--dry-run`, `--stage N`), `cli/chat.py` (REPL + /memory), `cli/memcheck.py` (멀티턴 기억 검증) 추가. orchestrator에 `PipelineTrace` 내장.

### 4. 백엔드 스켈레톤 타이핑 (Sonnet)

Opus가 확정한 구조 명세대로 Sonnet이 타이핑. 범위: 스텁(NotImplementedError) + schemas.py/allowlist 완전 구현 + 테스트.
