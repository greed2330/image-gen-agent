### 4. NSFW 태그 사전 + 시스템 프롬프트 설계 — 핵심 논의 (미결)

**오너 아이디어**: 포즈(체위), 삽입 묘사, 의상 상태, 표정 등 danbooru NSFW 태그를 카테고리별로 시스템 프롬프트에 미리 박아두면 → 모델이 추론 없이 룩업만 해도 되지 않나?

→ 방향은 정확함. 세부 설계는 Opus와 상의 예정.

**이 세션에서 나온 핵심 설계 인사이트**:

1. **상세 태그 사전은 저능 LLM을 구한다**: 태그 목록을 주면 LLM 임무가 "추론"에서 "매칭"으로 바뀜. 능력 저하된 abliterated 모델도 JSON 스키마 + 태그 목록이 있으면 충분히 동작 가능.

2. **파이프라인 레이어 분리가 진짜 해법**: 사용자 채팅 이해 문제는 모델 하나가 다 해결하는 게 아니라 역할 분리로 해결.
   - `qwen3:14b` (검열, 고성능) → 사용자 한국어 채팅 의도 파악, SFW 요소 구조화
   - `abliterated:14b` (무검열, 저능) → NSFW 태그 변환만 담당. 이미 파싱된 구조 입력받음

3. **설계 구멍 발견**: `qwen3:14b`는 검열 모델이라 NSFW 내용을 파싱하다가 거부할 수 있음.
   → **NSFW 레이어 분리 원칙이 07번 문서에 명문화되어 있지 않음.**
   → 해결 방향: SFW 파싱(스타일·인물·배경)은 qwen3:14b, NSFW 내용(행위·체위·의상·표정)은 abliterated가 직접 담당하도록 분리.

### 5. TIPO/DanTagGen 도입 결정 (Opus 재합류 후, 오너 제안)

**오너 질문**: "이미지 생성 프롬프트 합성 스페셜리스트 모델이 따로 없나?"

→ **있음. TIPO (KGen, ICLR 2026) / DanTagGen.** danbooru 태그 전용 학습 프롬프트 생성기.

**핵심 통찰 — "이해"와 "합성"은 별개 작업**:
- A. 이해(한국어→의도): 스페셜리스트 없음. 범용 LLM(qwen3:14b/클라우드/abliterated) 필수.
- B. 합성(씨앗태그→완성 프롬프트): **TIPO가 스페셜리스트.** danbooru 학습이라 조합·정렬·연관태그를 모델이 앎.

**왜 도입 (검증된 사실)**:
- TIPO와 Illustrious/NoobAI 둘 다 danbooru 태그 학습 → **같은 공용어. 포맷 정의 불필요.**
- 범용 LLM 어휘 부담↓ → hallucination 구조적 감소. + 기존 allowlist 검증 = 사실상 해결.
- ComfyUI 노드 존재.

**⚠️ 과신 경계 (이번 세션 교훈)**: 처음에 "TIPO가 NSFW native 처리·hallucination 0"이라 단정 → 오너 "팩트야?" 지적 → GitHub 원문 재확인. 결과: NSFW 품질·표기관례 일치는 **문서 미확인 = 미검증**. danbooru 학습 기반의 합리적 추론일 뿐. Phase 1/2 실측 전 확정 금지.

**시스템 프롬프트 위상 변화**: TIPO가 합성을 가져가면서 시스템 프롬프트는 "정수"에서 "부품"으로 격하. 정수는 "레이어 분리 + TIPO + allowlist" 아키텍처.

**문서 반영 완료**: 07(§1·§1-A·§2·§4), 03(합성 전용 모델 섹션), 01(③단계).
